# 기술 설명서

## 1. 클래스 책임 분리

이 프로젝트는 두 개의 클래스로 구성된다.

- **Quiz**: 퀴즈 한 문제의 데이터(문제, 선택지, 정답, 힌트)를 담고, 출력(`display`)과 정답 확인(`check_answer`)을 담당한다. 퀴즈 한 문제가 스스로 "자기를 보여주고" "정답을 판별하는" 책임만 가진다.
- **QuizGame**: 게임 전체 흐름을 관리한다. 퀴즈 목록, 점수, 히스토리 등의 상태를 보유하고, 메뉴 표시, 퀴즈 진행, 추가/삭제, 파일 저장/로드를 수행한다.

이렇게 분리한 이유는, Quiz가 "한 문제"의 개념이고 QuizGame이 "게임 시스템"의 개념이기 때문이다. Quiz는 자신이 어떤 게임에 속하는지 알 필요가 없고, QuizGame은 개별 문제의 내부 구조에 의존하지 않는다. 각자의 변경 이유가 다르므로 분리하는 것이 자연스럽다.

## 2. 로직 분리 기준

QuizGame 내부 메서드는 역할에 따라 세 가지로 나뉜다.

| 분류 | 메서드 | 역할 |
|---|---|---|
| 입력 처리 | `_get_number_in_range`, `_get_nonempty_input` | 사용자 입력을 받아 검증하고 유효한 값만 반환 |
| 게임 진행 | `play_quiz`, `add_quiz`, `show_quiz_list`, `show_score`, `delete_quiz` | 각 메뉴 기능의 흐름을 담당 |
| 데이터 저장 | `load_state`, `save_state` | state.json 파일 읽기/쓰기 |

입력 처리를 별도 메서드로 분리한 이유는, 검증 로직(범위 확인, 빈 입력 거부)이 여러 곳에서 반복되기 때문이다. `_get_number_in_range`는 메뉴 선택, 정답 입력, 문제 수 선택, 삭제 번호 선택 등 4곳에서 재사용된다.

## 3. state.json 읽기/쓰기 흐름

### 읽기 (load_state)

프로그램 시작 시 `QuizGame.__init__`에서 `load_state`를 호출한다. 세 가지 경우를 처리한다:

1. **파일 존재**: JSON을 파싱하여 quizzes, best_score, history 등을 복원한다.
2. **파일 미존재** (`FileNotFoundError`): 첫 실행이므로 기본 퀴즈 5개로 초기화한다.
3. **파일 손상** (`JSONDecodeError`, `KeyError`, `TypeError`): 안내 메시지 출력 후 기본 데이터로 복구하고 즉시 `save_state`를 호출하여 손상된 파일을 덮어쓴다.

### 쓰기 (save_state)

상태가 변경되는 모든 시점에서 즉시 `save_state`를 호출한다 (eager save 방식):

- 퀴즈 풀기 완료 후 (`_record_result`)
- 퀴즈 추가 후 (`add_quiz`)
- 퀴즈 삭제 후 (`delete_quiz`)
- 파일 손상 복구 시 (`load_state`)
- Ctrl+C/EOF 종료 시 (`run`의 except 블록)

이 방식은 "종료 시 한 번 저장"보다 데이터 유실 위험이 적다. 프로그램이 비정상 종료되어도 마지막 변경까지 저장되어 있다.

## 4. 안전 종료 처리

`run` 메서드의 메인 루프를 `try/except`로 감싸서 `KeyboardInterrupt`(Ctrl+C)와 `EOFError`(Ctrl+D)를 포착한다. 포착 시 종료 메시지를 출력하고 `save_state`를 호출하여 현재 상태를 저장한 후 프로그램을 종료한다.

추가로, `_record_result`에서는 상태 변경(history 추가, best_score 갱신)을 print() 호출 없이 연속으로 수행한 뒤 `save_state`를 호출하고, 그 이후에 결과를 출력한다. 이는 print()가 시스템 콜을 동반하여 그 사이에 KeyboardInterrupt가 발생할 확률이 높기 때문이다. 상태 변경 사이에 I/O가 끼면 논리적으로 불일치한 상태(예: history는 추가됐지만 best_score는 미갱신)로 저장될 수 있다.

## 5. 커밋 단위 및 메시지 규칙

### 커밋 단위

하나의 커밋은 하나의 완결된 변경을 담는다. "기능 추가", "리팩터링", "문서 작성"처럼 변경의 성격이 다르면 별도 커밋으로 분리한다. 여러 파일을 수정하더라도 하나의 목적에 속하면 한 커밋에 포함한다.

### 메시지 규칙

`타입: 설명` 형식을 사용한다.

| 타입 | 용도 |
|---|---|
| Feat | 새로운 기능 추가 |
| Fix | 버그 수정 |
| Refactor | 동작 변경 없는 구조 개선 |
| Docs | 문서 작성 및 수정 |

설명은 한국어로 작성하며, "무엇을 했는가"를 간결하게 적는다.

## 6. 클래스 사용 이유

함수만으로도 이 프로그램을 구현할 수 있다. 하지만 함수만 사용하면 quizzes, best_score, history, has_played 등의 상태를 전역 변수로 관리하거나 함수 간에 매번 인자로 전달해야 한다. 상태가 늘어날수록 어떤 함수가 어떤 데이터에 접근하는지 추적하기 어려워진다.

클래스를 사용하면 관련 데이터와 그 데이터를 조작하는 함수를 하나로 묶을 수 있다. `self`를 통해 상태에 접근하므로 함수 시그니처가 단순해지고, 데이터와 동작의 관계가 명시적이다. 예를 들어 `self.save_state()`는 "이 게임 인스턴스의 상태를 저장한다"는 의미가 코드 자체로 드러난다.

## 7. JSON 저장 이유 및 특징

### 선택 이유

- Python 표준 라이브러리(`json` 모듈)만으로 읽기/쓰기가 가능하다. 외부 의존성이 없다.
- 사람이 직접 읽고 수정할 수 있다. 디버깅이나 데이터 확인이 쉽다.
- Python의 dict, list와 직접 변환된다. 별도의 변환 로직이 필요 없다.

### JSON 형식의 특징

- 키-값 쌍으로 구조화된 데이터를 표현한다.
- 문자열, 숫자, 불린, 배열, 객체, null을 지원한다.
- 언어에 독립적인 텍스트 형식이라 Python 외 다른 도구에서도 읽을 수 있다.

## 8. try/except 필요성

파일 I/O는 프로그램 외부 자원(파일 시스템)에 의존하므로, 코드가 올바르더라도 런타임에 실패할 수 있다. 이 프로젝트에서 처리하는 예외는 다음과 같다:

| 예외 | 발생 상황 | 처리 |
|---|---|---|
| `FileNotFoundError` | 첫 실행 시 state.json이 없음 | 기본 퀴즈로 초기화 |
| `json.JSONDecodeError` | 파일 내용이 유효한 JSON이 아님 | 안내 후 기본 데이터로 복구 |
| `KeyError` | JSON에 필요한 키가 없음 | 안내 후 기본 데이터로 복구 |
| `TypeError` | 값의 타입이 예상과 다름 | 안내 후 기본 데이터로 복구 |
| `ValueError` | 사용자가 숫자 대신 문자를 입력 | 재입력 요청 |

try/except 없이 이 예외들이 발생하면 프로그램이 즉시 종료되어 사용자가 오류 메시지만 보게 된다. 예외 처리를 통해 프로그램이 중단 없이 적절한 대응(복구, 재입력 요청)을 할 수 있다.

## 9. 브랜치와 병합의 의미

### 브랜치

브랜치는 main(주 코드)에 영향을 주지 않고 독립적으로 작업할 수 있는 공간이다. 예를 들어 `feature/safe-exit` 브랜치에서 안전 종료 기능을 수정하는 동안, main 브랜치의 코드는 변하지 않는다. 수정이 완성되지 않은 상태에서 main이 깨지는 일이 없다.

### 병합

작업이 완료된 브랜치를 main에 병합(merge)하여 변경 사항을 통합한다. `--no-ff` 옵션으로 병합하면 병합 커밋이 생성되어, git log 그래프에서 "어떤 브랜치에서 작업한 뒤 합쳤다"는 이력이 시각적으로 남는다.

### 협업에서의 의미

여러 사람이 동시에 다른 기능을 개발할 때, 각자 브랜치에서 작업하면 서로의 코드에 간섭하지 않는다. 완성된 기능만 main에 병합하므로 main은 항상 동작하는 상태를 유지할 수 있다.

## 10. 데이터 구조 설계 이유

state.json의 구조는 다음과 같다:

```json
{
    "quizzes": [...],
    "best_score": {"score": 100, "correct": 3, "total": 3},
    "has_played": true,
    "history": [...]
}
```

- **quizzes 배열**: 퀴즈는 순서가 있고 개수가 변하므로 배열이 자연스럽다. 각 항목은 question, choices, answer, hint를 가진 객체로, Quiz 클래스의 속성과 1:1 대응된다.
- **best_score 객체**: score, correct, total을 하나로 묶었다. 이 세 값은 항상 함께 사용되므로 별도 최상위 키로 분리하면 관계가 흐려진다.
- **has_played 불린**: 한 번도 플레이하지 않은 상태와 점수가 0인 상태를 구분하기 위해 필요하다. 이 값이 없으면 best_score가 0일 때 "아직 안 풀었음"과 "풀었는데 전부 틀림"을 구분할 수 없다.
- **history 배열**: 시간순 기록이므로 배열이 적합하다. 각 항목은 날짜, 문제 수, 정답 수, 점수를 포함한다.

## 11. JSON 저장 방식의 확장성 한계

현재 이 프로젝트는 소규모 데이터(퀴즈 수십 개, 게임 기록 수십 건)를 단일 사용자가 사용하므로 JSON 파일 저장이 적합하다. 하지만 퀴즈가 1000개 이상으로 늘어나면 다음과 같은 한계가 발생한다:

- **전체 로드/저장**: 퀴즈 하나를 추가해도 전체 파일을 읽고 다시 쓴다. 데이터가 커질수록 매 저장마다 성능이 떨어진다.
- **부분 수정 불가**: "3번 퀴즈만 수정"이 불가능하다. 항상 전체를 덮어쓴다.
- **동시 접근 불가**: 여러 프로세스가 동시에 같은 파일을 읽고 쓰면 데이터가 손상될 수 있다.
- **인덱스 없음**: 특정 조건의 퀴즈를 찾으려면 전체를 순회해야 한다.

이런 한계가 실제 문제가 되는 시점에서 데이터베이스 도입을 고려할 수 있다. 현재 규모에서는 JSON의 단순함이 장점이다.

## 12. 요구사항 변경 시 수정 범위

### 채점 방식 변경 (예: 난이도별 차등 점수)

1. **Quiz 클래스**: 난이도 속성 추가 (`self.difficulty`)
2. **`_ask_single_question`**: 난이도에 따라 다른 점수를 반환하도록 수정
3. **`_record_result`**: 점수 계산 방식 변경에 맞게 조정
4. **`save_state`/`load_state`**: 새 속성의 직렬화/역직렬화 추가
5. **DEFAULT_QUIZZES**: 기존 퀴즈에 난이도 값 추가

### 퀴즈 구조 변경 (예: 선택지 개수 가변)

1. **Quiz 클래스**: choices 길이를 4로 고정하지 않도록 수정
2. **`_ask_single_question`**: 정답 범위를 `1~len(choices)`로 동적 변경
3. **`add_quiz`**: 선택지 입력 루프를 가변 개수에 맞게 수정
4. **state.json**: 구조 자체는 이미 배열이므로 스키마 변경 불필요

두 경우 모두 QuizGame의 메뉴 표시(`show_menu`), 목록 보기(`show_quiz_list`) 같은 기능은 수정할 필요가 없다. 변경이 필요한 범위가 특정 메서드로 한정되는 것은 클래스 책임 분리의 효과이다.
